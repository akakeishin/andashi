## 1. 失敗パターン

| 失敗 | 症状 | 修正 |
|---|---|---|
| 形容詞の盛り付け | 革新的、没入型、心に響く、唯一無二 | 主体、行動、変化、代償へ書き換える |
| 疑似多様性 | アプリ、サイト、コミュニティで同じ機構 | 主要機構を別々にする |
| 機能の束 | AI、通知、ポイント、共有を並べる | 一つの行動仮説を中心に再構成する |
| ランダムな奇抜さ | 変な名前、小物、演出だけが違う | その要素が選択と結果をどう変えるか要求する |
| 早すぎる収束 | 最初の案を詳しくして採用する | 独立バッチを作ってから比較する |
| 後付け選抜 | 勝者は美点、敗者は別々の弱点で説明する | 依頼由来の共通条件を先に固定し、全候補へ同じ順序で適用する |
| 弱点の儀式化 | 「コストが課題」のような一般論 | この案が最も壊れやすい場面を特定する |
| 質問過多 | 依頼に答えず長い質問票を返す | 仮定を置いてスターター案を出す |
| 偽の確信 | 効果、需要、空き名称を断定する | 仮説と確認方法を示す |
| 過剰な表 | すべてを採点し精密に見せる | 結論を変える差だけ見せる |
| 既存案の無視 | Refineで核まで作り替える | Keep / Reject / Openを先に固定する |
| 自己監査の演出 | 自分で作った案へ高得点を付け「検証済み」と呼ぶ | 二値gate、別文脈の反証、人間Commit、現実のprobeへ分ける |
| 形式だけの独立性 | 同じ文脈で役名を変え、独立評価と呼ぶ | 別文脈の有無を区別し、使えなければ正直な縮退にする |
| probeの量産 | 全案が「2週間パイロット」「アンケート」 | 各案で最大の未知を先に特定し、最小の反証方法を変える |

### Muse固有の失敗

| 失敗 | 症状 | 修正 |
|---|---|---|
| 無目的の空白 | 「自由に詩的に」だけで月、雨、記憶、影へ戻る | ユーザーの語・音・字形・文骨格をDonorにし、操作族を指定する |
| シュール名詞の山 | 奇妙な名詞を三つ以上並べ、関係も痕跡もない | 普通の場面へ異常を一つだけ置き、具体的な結果を残す |
| 一構文の量産 | 全行が「XがYを忘れる」「XはYの味」 | 音、統語、手続き、時制、欠落へ操作族を切り替える |
| 早すぎる意味閉鎖 | 「これは孤独の象徴」「テーマは再生」 | メタ散文を削り、像と文だけで終える |
| 隠れた商品化 | 断片ごとに用途、ターゲット、企画価値を付ける | ユーザーがHarvestするまでShapingを起動しない |
| 偽のHarvest | 一発二段依頼でモデルの好みをユーザー選択として扱う | 純Museへ入れず、明示委任時だけmodel-selected donorとして逐語表示する |
| CTAによる収束 | 末尾でおすすめや「形にしますか」を求める | 指定形式の断片で静かに停止する |
| 参照不能な複数断片 | 指定数は合うが番号がなく、後続で対象を安定して指せない | 別形式指定がなければ1から指定数まで欠番なく連番にする |
| Donor消失 | 別の入力でも同じ断片が出せる | 入力の語、音、文骨格、物性を追跡できる形へ作り直す |

## 2. 分野別の早期定番

次の語や型は自動失格ではない。これは見落としを拾う補助smokeであり、発散前の動的な最頻回答baselineの代わりではない。`divergence-lenses.md` に従い、毎回ブリーフ固有の機構・形態クラスを先に作る。下の語がなくても一歩ずらした同族なら同じ定番として扱う。

- 商品・機能: AI化、ダッシュボード、パーソナライズ、通知、連続記録、バッジ
- コンテンツ: 舞台裏、UGC、インフルエンサー、ハッシュタグ、バズ、限定配信
- イベント: 基調講演、パネル、交流会、フォトスポット、ノベルティ
- 業務改善: 自動化、一元化、可視化、チャットボット、全部入りSaaS
- ネーミング: AI、Hub、Plus、ミライ、つながる、同一接尾辞の量産
- 物語: 選ばれし者、記憶喪失、謎の組織、実は闇を抱えた可愛い存在

王道を求められた場合は型を壊す必要はない。王道の約束を守りつつ、対象固有の制約、選択、代償、表現で具体化する。

## 3. 修正テスト

### MuseのDonor消去テスト

入力の固有語を別の素材へ交換したとき、ほぼ同じ出力が成立するなら、素材でなくモデル既定の詩語から生成している。ランタイムで美しさを採点せず、別の操作族を使い、Donorの音・統語・物性の痕跡を戻す。

継続評価では複数Donorの出力を匿名化し、人間または別評価文脈が元Donorを対応づけられるかを見る。対応できない傾向は一般化でなく素材非依存の疑いとして扱う。

### 一異常テスト

断片から奇妙な要素を一つ残してほかを日常語へ戻す。それでも像が動くなら関係がある。全要素を奇語にしないと成立しない場合は、異常の結果を具体化して作り直す。

### レジスター偏りテスト

バッチの大半が憂愁、郷愁、夢、リミナルな語調に戻っていないかを見る。同時に、語調が違っても具体物・人物・手続きが同じ連想場へ集中していないかを見る。偏った場合は「もっと独創的に」と命じず、Donorを保ったまま検品、料理、配管、会計、身体動作など別のレジスターと連想場を引く。

### メタ散文除去テスト

「意味」「象徴」「狙い」「おすすめ」「弱点」「使い道」の説明節をすべて消す。断片そのものが残らないなら、Museではなく解説を生成していた。

### 保存破壊テスト

各変奏について、入力から何を一つ保存し、どの関係を一つ壊したかを内部で指せるか確認する。保存が「雰囲気」、破壊が「奇妙にする」しか言えない断片は作り直す。ShapingではHard条件を壊す側へ置かない。

### 名詞交換テスト

業界名、対象者名、固有名詞を別の語へ交換しても成立する文章は一般論である。交換できない具体的な利用場面、制約、行動を足す。

### 「だから何が起きる」テスト

目立つ要素ごとに、直接の効果と、その次の反作用を答える。二段目がない要素は装飾の可能性が高い。

### 三つの証拠テスト

案が次のうち三つを生めるか確認する。

- 実際の利用・運用場面
- 失敗または葛藤場面
- 継続・再利用場面
- 最小の試験
- 案固有の代償

### 反対者テスト

最も困る人、採用しない人、誤用する人を一人ずつ想定する。反対がすべて「理解不足」で片づく案は、利害と代償の検討が弱い。

### 無申告の定番テスト

候補に早期定番の語や機構が含まれるのに、既知パターンとして扱っていない場合は再検査する。モデルが `known_pattern` と自己申告した候補だけを調べる方式にしない。定番を残すなら、対象固有の逸脱と二つ以上の下流効果を言える状態にする。

### 反証先行テスト

候補を改善する前に、凍結した版で最初の破断点を特定する。批評中に文章を直して弱点が最初からなかったように見せない。修正版は新しい候補として再検査する。

## 4. 短い校正例

### 商品

弱い: 「AIが共有工具の利用を最適化し、通知とバッジで返却を促す。」

補正: 「借り手を追いかけるのではなく、工具が棚へ戻るまで次の貸出札を外せない。木札と現物5点だけで、返却漏れと取り違えが減るかを一日試す。」

差はAIの有無ではなく、介入点と行動仮説にある。

### 企画

弱い: 「地域の舞台裏をSNSで発信し、参加型キャンペーンにする。」

補正: 「閉店後に一つだけ残る作業を店主が30秒で見せ、翌朝に客がその作業の結果を探せる店頭札を置く。視聴ではなく来店時の発見へつなぐ。」

差は動画の題材ではなく、オンラインから現地行動への機構にある。

### 業務改善

弱い: 「申請を一元化し、ダッシュボードで可視化する。」

補正: 「申請後の追跡ではなく、差し戻し理由上位3件を入力時に解消する。新システムを増やさず、既存フォームの順序変更を2週間だけ試す。」

差は画面ではなく、ボトルネックへ介入する時点にある。

### キャラクター

弱い: 「かわいいが実は闇を抱え、雨の日だけ透けるマスコット。」

補正: 「人の失くし物を一晩だけ自分の記憶として預かる。返すほど自分の名前を忘れるため、助けたい欲望と存在を保つ選択が毎回衝突する。」

差は奇妙な見た目ではなく、規則、選択、代償から場面が生まれる点にある。
