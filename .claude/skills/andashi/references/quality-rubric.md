# 反証と比較の規律

この資料は採点表ではない。生成モデルの数値自己採点を品質証明にせず、粗い失敗を落とし、残る価値衝突を理由付きで見えるようにする。

Quick / Studio / Refine / Decide / RescueのShapingだけに適用する。Museでは目的、候補差、案固有の失敗、実現性、probeを検査しない。安全、事実、権利、秘密、ユーザー指定の形式はMuseでも別の普遍境界として守る。

## 1. Gatekeeper: 合否だけを見る

次のどれかに違反する候補は比較前に落とす。

- 明示されたHard条件、禁止事項、個数、媒体、期限、予算に違反する。
- 目的、対象、主要な一手のいずれかが空である。
- 未確認の需要、効果、価格、法的安全性、商標、空き名称、競合、流行を事実として断定する。
- 実行に不可欠な資源、権利、データ、承認、運用主体を隠す。
- 別候補と主要機構が同じで、名前・媒体・表現だけが違う。
- 欺瞞、強制、同意侵害、差別、安全上の重大な外部性を隠す。
- 案固有の失敗場面または最大の未知を示せない。

失格を合計点で相殺しない。全案失格も有効な結果である。

## 2. 差異を検査する

候補名と文体を隠し、次を比べる。

1. 主体または影響を受ける人
2. 介入する時点
3. 行動・選択を変える主要機構
4. 生じる価値または体験
5. 負担、危険、自由、時間、費用を支払う人
6. 最大の未知を確かめる方法

主要機構が同じ、または主体・介入点・主要機構・価値交換・代償のうち異なる軸が2つ未満なら、別案と数えず統合または再探索する。結果やprobeだけの差、文章類似度、候補名は機構距離の証拠にしない。

差異の証拠が書かれていない場合は「重複なし」とみなさず、「比較不能」として戻す。証拠欠落を合格扱いしない。

## 3. Challenger: 案を直さずに壊す

生存案ごとに次を答える。

### 最初の破断点

「コストが課題」のような一般論ではなく、誰が、どの場面で、何を理由に止めるかを書く。壊れた後に誰へどんな負担が移るかまで追う。

### 合理的な反対者

採用しない人を「理解不足」で片づけない。その人の目的、現行代替、失うものを示す。反対が合理的なら、採用対象を狭めるか案を落とす。

### 装飾除去

固有名詞、AI、演出、媒体、可愛い小物を外し、残る一手を一文にする。残らなければ独自性ではなく装飾である。

### 反証probe

各案で最も結論を変える未知を一つ選ぶ。probeは、その未知を最小コストで否定できる方法にする。

probeを書く前に手法族を一つ付ける。たとえば現場観察、欠損停止、紙・人力代替、遅延再生、対比較・おとり、運用痕跡計数、想起、公式資料確認、専門家確認である。語尾や期間が違っても同じ行為なら同族とする。最大の未知が異なる候補へ同じ手法族を3件以上使わない。同一の未知を条件だけ変えて統制比較する場合は、同族反復の理由を明記する。

悪い例:

- 2週間パイロットを行う
- ユーザーに聞く
- KPIを見る

良い例:

- 紙のカードだけで5人に選択してもらい、説明なしで意図した行動が3人以上に起きるかを見る
- 会議を1回だけ止め、代替問い合わせ・遅延・未決定を数える
- 候補名を音声だけで10人に伝え、24時間後の再生と誤綴りを確認する

候補間で未知が違うなら、probeの方法も変える。全案を同じ期間・同じアンケート・同じ無記名投票へ押し込まない。手法族を増やすこと自体が目的ではなく、結論を変える未知に合う観測を選ぶ。

### 外部性

成功した場合にも、負担、排除、監視、誤用、責任が別の人へ移らないかを見る。利益を受ける人と費用を払う人が違う場合は明示する。

## 4. 既知パターンを扱う

定番は自動失格ではない。次の三つが揃う場合だけ独立候補として残す。

1. 使っている既知パターン
2. 対象固有の具体的な逸脱
3. 逸脱が行動、選択、運用、感情へ生む二つ以上の下流効果

候補文中に「AI」「ダッシュボード」「通知」「ポイント」「UGC」「インフルエンサー」「パネル」「交流会」「自動化」「一元化」「Hub」など早期定番があるのに、既知パターンとして認識していない場合は自己申告を信用せず再検査する。

## 5. 理由付きペア比較

意思決定が必要なときだけ比較する。絶対点を付けない。まずブリーフから共通比較条件を2〜4個固定し、全候補へ同じ順序で一度適用する。その後、結論を分ける候補だけを二つずつ比較する。

```text
共通比較条件: C1 / C2 / C3
候補A: 各条件の証拠 / 残る条件 / 落ちる条件
候補B: 各条件の証拠 / 残る条件 / 落ちる条件
候補C: 各条件の証拠 / 残る条件 / 落ちる条件
```

候補固有の弱点、未知、probeは必要だが、それぞれ別の物差しで比較した証拠にはしない。残す候補の美点だけ、落とす候補の負担だけを列挙する後付け選抜は禁止する。

```text
比較するユーザー条件:
Aが勝つ条件と根拠:
Bが勝つ条件と根拠:
現在のブリーフではどちらか:
結論を反転させる未知:
```

候補順を反転して同じ証拠で再確認する。順序で結論が変わるなら推奨を保留する。最高点ではなく、ユーザーが重視する制約と許容する代償に最も合う案を選ぶ。指定数へ絞る場合は、全候補に適用した共通条件を表示してから、残す理由と落とす理由を同じ条件の言葉で書く。

## 6. 分野別の問い

主分野の三つだけを追加する。

| 分野 | 問い1 | 問い2 | 問い3 |
|---|---|---|---|
| 商品・サービス | 現行代替より何が変わるか | 導入・継続行動は現実的か | 技術・運用・資源へ合うか |
| 企画・コンテンツ | 対象者の緊張を捉えるか | チャネルから目的行動へつながるか | 反復・制作を続けられるか |
| ネーミング | 意味とトーンへ合うか | 音・表記・記憶に耐えるか | 誤解・衝突・未調査を扱うか |
| イベント・体験 | 参加前後を含め何が変わるか | 物流・安全・アクセシビリティに耐えるか | 参加の機構があるか |
| 業務改善 | 症状でなく原因へ触れるか | 戻せるか、統制を保てるか | 効果と副作用を観測できるか |
| 物語・創作 | 三つ以上の異なる場面を生むか | 感情・テーマの圧力が選択へ現れるか | 媒体・規模で表現できるか |
| 一般 | 依頼固有の成功条件へ直結するか | 実行または検証経路があるか | 影響を受ける第三者を扱うか |

## 7. 停止と再探索

- 全案が同じ問いで壊れるなら、その問いを新しい機構スロットにして再探索する。
- 一案が明確に優位なら、弱い三案目を足さない。
- 探索目的なら、勝者を作らず異なる価値衝突を残す。
- 新しい案より人間のCommitまたは現実のprobeの情報価値が高ければ止める。
- Challengerが修正版を思いついても、その場で評価対象を変えない。新IDでGatekeeperへ戻す。
