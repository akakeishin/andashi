# 出力形式

結論または使える候補から始める。工程報告を先頭にしない。ユーザーの個数、長さ、形式、言語を優先する。ただし安全、未確認事実、法的・専門的な留保は、短さのために消さない。

万能テンプレートは使わない。MuseとShapingを同じ形式へ押し込まず、依頼に必要なものだけを選ぶ。

## 1. Muse: 断片だけを残す

ユーザー指定の形式があればそれだけを使う。未指定なら参照用の番号と断片本文だけを返し、複数断片では `1.` から指定数まで欠番なく付ける。「断片だけ」「前置きなし」は番号なし指定とみなさない。

```text
1. 断片

2. 断片
```

番号は順位ではない。前提、操作名、テーマ、象徴、意味、弱点、使い道、probe、次の一手を表示しない。タイトルも、タイトル自体が依頼された場合だけ付ける。

末尾に `どれが好きですか`、`形にしますか`、操作メニューを自動で置かない。指定数で静かに終える。ユーザーが特定断片から続けた場合は、その本文をDonorにして次の変奏だけを返す。

ユーザーが明示的にHarvestした場合は、Muse出力へ説明を後付けせず、次のShaping応答でだけ短く境界を示す。

```markdown
残す断片: ユーザーが選んだ本文を逐語で
形にする先: ユーザーが指定した成果物
説明で消さないもの: ユーザーが残したいと示した感触
```

その後に初めて、該当するQuick / Studio / Refine形式を使う。

一つの回答で「断片を出し、あなたが一つ選び、成果物にする」と明示された場合は純Muse形式を使わない。Shapingの生成前置きとして指定数の断片を出し、`モデルが選んだDonor: <逐語引用>` と明示してから成果物へ進む。`ユーザーが選んだ`、`Harvestした`とは書かず、引用本文を都合よく改変しない。

## 2. Quick: 短い機構ポートフォリオ

```markdown
前提: 結論を変える仮定だけ

1. 案名 — 誰に何をする案か。差は機構。弱点は代償。最初はprobeで確かめる。
2. …

次に見るなら: 最も判断を変える一点
```

各案の一行に、他案との差、案固有の弱点、次の一手を圧縮する。指定がなければ3〜5案。ユーザーが1案なら1案、2案なら2案だけを返す。100案なら意味領域や機構スロットで群分けし、水増しを避ける。

## 3. Studio Stage A: Commit用ポートフォリオ

磨く前の異なる賭けを見せる。

```markdown
## 今回の前提
- Goal / Hard / 仮定のうち結論を変えるもの

## 方向
### P1 — 一文の核
- 一手: 行動や体験を変える機構
- 支払うもの: 最大の代償

### P2 — …

選ぶ / 殺す / 曲げる / 任せる、のどれでも続けられます。
```

候補が1案でも4操作をすべて表示する。Stage Aかauto-commitかは `studio-protocol.md` の決定表で先に分岐し、この資料では再定義しない。

完成度の高い説明で弱い機構を有利にしない。ユーザーへは一つの判断だけ求める。「どれが好きか」より、許容する代償や守る価値を聞く。

## 4. Stage B / Decide: 生存案の反証と具体化

```markdown
## 結論
推奨または、順位を付けない理由

## 生存案
### 案名 — 一文の核
- 成立機構:
- proof-of-life:
- 最初に壊れる場面:
- 最大の未知:
- 次のprobe:

## 比較
共通比較条件を2〜4個固定する。

| 候補 | 条件1の証拠 | 条件2の証拠 | 残る／落ちる条件 |
|---|---|---|---|
| A | … | … | … |
| B | … | … | … |

必要なら境界にいる二案だけ、どの条件ならA／Bが勝つかを追加する。

## 次の一手
最も情報価値の高い行動
```

比較の適用順、探索時の非選抜、全案失格は `quality-rubric.md` を正本とする。この節は、そこで得た判断を同じ欄へ表示する形だけを示す。

## 5. Refine / Rescue

```markdown
## 守る核
- Fixed / Keep

## 詰まり
- 症状でなく、弱い機構または制約衝突

## 改訂方向
- 核を保つ
- 同じ目的を別機構で達成する
- 壊してよい前提を一つ変える

## 台帳
Fixed: … / Keep: … / Reject: … / Open: …
```

ユーザーが嫌った方向を名前だけ変えて復活させない。台帳は更新があるときだけ短く表示する。

## 6. ネーミング

候補名より先に命名戦略の差を示す。

```markdown
## 領域A — 狙う印象と言語機構
- 候補（読み）: 意味 / 使用場面 / 誤読・弱点

## 領域B — …

## ショートリスト
## 未確認
- 商標、ドメイン、SNS名、他言語含意
```

一案だけ求められても、内部またはScout間では複数の意味領域を比べてから一案を返す。空きや法的利用可否は調査なしに断定しない。

## 7. 物語・キャラクター

```markdown
### 案名 — 一文の核
- 欲望または中心目的:
- 規則・関係の圧力:
- 選択と代償:
- proof-of-lifeとなる一場面:
- 破断テスト: 日常 / 圧力 / 親密さ・喪失
- 枯れやすい弱点:
```

設定資料の量ではなく、異なる場面と選択を生む力を示す。世界観なら規則、価格、分配、悪用、日常への影響を加える。

## 8. 一発完結時の透明性

Studio相当を一つの応答で完結し、サブエージェントや人間Commitを使えなかった場合だけ、必要に応じて次を短く添える。

> 選抜と弱点分析は同じ会話文脈内の検査です。最終判断は上記probeで確かめてください。

軽いQuickで毎回この注記を出して会話を重くしない。高コスト、高リスク、「徹底的」「検証済み」を求められた場合に透明性を優先する。
